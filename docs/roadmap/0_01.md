# v0.01

## Cross-platform binaries via bblanchon/pdfium-binaries

**In progress.**

> Plan at
> [`../plans/plan_bblanchon_binaries.md`](../plans/plan_bblanchon_binaries.md).

The original plan was to build PDFium from source via a bespoke `pdfium-build`
CI pipeline. This was superseded after diagnosing two blocking problems: the
iOS static xcframework dead-strips all PDFium symbols beyond
`FPDF_InitLibraryWithConfig` (making every API call fail at runtime), and the
home-built binaries are excessively large (iOS ~258 MB static `.a` vs 6.5 MB).

The replacement approach adopts
[bblanchon/pdfium-binaries](https://github.com/bblanchon/pdfium-binaries) —
a community-tested set of pre-built PDFium releases that cover every target
platform. This eliminates the build pipeline entirely.

**Target platforms:**

| Platform       | Library form          | Status                                                   |
| -------------- | --------------------- | -------------------------------------------------------- |
| macOS arm64    | dylib                 | Shipping (our build); migrating to bblanchon             |
| Linux x86_64   | shared lib            | Shipping (our build); migrating to bblanchon             |
| Linux arm64    | shared lib            | Shipping (our build); migrating to bblanchon             |
| iOS arm64      | dylib (xcframework)   | Broken (dead-strip); fix via bblanchon plan              |
| Android arm64  | shared lib            | Untested; fix via bblanchon plan                         |
| Android x86_64 | shared lib            | Untested; fix via bblanchon plan                         |
| Windows x86_64 | DLL                   | Future — bblanchon provides; out of scope for this plan  |
| Web (WASM)     | .wasm + .js           | Future — bblanchon provides; blocked on WASM backend     |

## Mobile integration test app (iOS and Android)

**Complete (2026-06-26).**

> Implemented in
> [`../plans/completed/plan_mobile_integration_tests.md`](../plans/completed/plan_mobile_integration_tests.md).

The desktop test suite (`dart test`) cannot exercise the iOS and Android
library-load paths — `DynamicLibrary.process()` (iOS static xcframework) and
bare-name `DynamicLibrary.open('libpdfium.so')` (Android) are platform-gated
branches that no host test can reach. This work creates a standalone Flutter
integration test app in `integration_test_app/` that runs the equivalent of the
desktop suite on a connected device or emulator.

**Delivered:**

- `integration_test_app/` — standalone Flutter package with 11-group integration
  test suite (Smoke, Page count, Metadata, Plain text, Rendering, Annotations,
  Image extraction, TOC, Search, Thumbnail, Error handling + Page size).
- `fetch_mobile_binaries.sh` downloads and SHA-256-verifies iOS xcframework and
  Android `.so` files from `version_pdfium.json` (single source of truth).
- `version_pdfium.json` extended with `ios-arm64`, `android-arm64`,
  `android-x64` entries; `update_pdfium_manifest.sh` updated accordingly.
- iOS: two-target local SPM package (`binaryTarget` + C anchor source target)
  prevents linker dead-stripping of the static xcframework.
- Android: `.so` in `jniLibs/`; loaded with `DynamicLibrary.open('libpdfium.so')`.
- Makefile targets: `sync_fixtures`, `fetch_mobile_binaries`, `ios_test`,
  `android_test`, emulator lifecycle targets.
- `make pre_commit` passes (398 tests, zero analyzer issues).

**Device verification status:**

| Platform       | Artifact state       | Test status                      |
| -------------- | -------------------- | -------------------------------- |
| iOS arm64      | xcframework shipping | Pending — run `make ios_test`    |
| Android arm64  | .so shipping         | Pending — run `make android_test`|
| Android x86_64 | .so shipping         | Pending — run `make android_test`|

On-device verification requires a connected device or booted emulator and is
performed manually with `make ios_test` / `make android_test`. CI automation is
out of scope for this milestone.

## Test coverage: 95 %

**In progress.**

> Plan at
> [`../plans/plan_test_coverage_95.md`](../plans/plan_test_coverage_95.md).

The `dart test` suite currently covers 61.4 % of `betto_pdfium` (1,100 / 1,792
tracked lines). The target is 95 %. Four major public API methods —
`extractAnnotations()`, `getDocumentInfo()`, `renderPageToBytes()`, and
`getPageSize()` — have zero coverage in the desktop test suite, together
accounting for the majority of the gap. The plan addresses this in four phases:
pure-Dart value-type unit tests, integration tests for the missing API methods,
error-path tests, and isolate infrastructure edge cases.
