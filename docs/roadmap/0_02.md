# v0.02

This roadmap picks up platform-coverage work deferred from v0.01. Both
bblanchon/pdfium-binaries **already ships** pre-built artifacts for Windows and
WASM — there is no build pipeline to construct. The work is wiring those
existing binaries into the distribution contract and implementing the runtime
backends.

## Windows x86_64

**Status: Complete.**

> Plan at [`../plans/completed/plan_windows_support.md`](../plans/completed/plan_windows_support.md).

bblanchon/pdfium-binaries publishes `pdfium-win-x64.tgz` (containing
`bin/pdfium.dll`) alongside every chromium/NNNN release. No separate build
pipeline is required — the binary already exists.

The work is mechanical: wire the DLL into the existing distribution
infrastructure and implement the Windows library-load path in the runtime.

**Scope:**

- Add `windows-x64` to `version_pdfium.json` and update
  `scripts/update_pdfium_manifest.sh` to download and checksum
  `pdfium-win-x64.tgz`.
- Update `scripts/fetch_pdfium.sh` to handle the `MINGW`/`CYGWIN`/Windows OS
  case and install `pdfium.dll` to `third_party/pdfium_bin/windows_x64/`.
- Update `hook/build.dart` to emit a `CodeAsset` for `pdfium.dll` on
  `OS.windows`, mirroring the existing macOS/Linux paths.
- Update `pdfium_isolate.dart`:
  - `_defaultDylibPathOrNull()` — probe the legacy
    `third_party/pdfium_bin/windows_x64/pdfium.dll` path.
  - `_openLibrary()` — add `Platform.isWindows` →
    `DynamicLibrary.open('pdfium.dll')`.
- Update `spec/01_binary_distribution.md` to document the Windows artifact and
  installed layout.
- Run `make pre_commit` to verify all existing tests still pass.

**Testing constraint:** end-to-end test execution requires a Windows machine
(physical or GitHub Actions `windows-latest` runner). The mechanical changes
above can be authored and reviewed on macOS/Linux, but final verification
cannot proceed without a Windows environment. This is tracked as a
prerequisite in the plan.

## Web (WASM)

**Status: Complete.**

> Plan at [`../plans/completed/plan_wasm_support.md`](../plans/completed/plan_wasm_support.md).

bblanchon/pdfium-binaries publishes `pdfium-wasm.tgz` (containing
`lib/libpdfium.wasm` + `lib/libpdfium.js`) alongside every chromium/NNNN
release. No Emscripten build is required — the binary already exists.

The work has two distinct parts: (1) **binary distribution** — getting the WASM
artifact into the manifest and into the browser, and (2) **runtime
implementation** — replacing `_document_web.dart`'s stubs with real
`dart:js_interop` calls into the Emscripten module.

**Scope:**

- Extend `version_pdfium.json` schema to support the dual-file WASM artifact
  (`libpdfium.wasm` + `libpdfium.js`) and add a `wasm` platform entry.
- Update `scripts/update_pdfium_manifest.sh` to download and checksum
  `pdfium-wasm.tgz`.
- Decide and document the web distribution mechanism: how do the WASM + JS
  files reach the browser at runtime? (Options: Flutter web asset, CDN URL
  supplied by caller, `fetch_pdfium.sh`-style developer download.) See plan
  for decision.
- Implement `lib/src/document/_document_web.dart` using `dart:js_interop`:
  - Load and initialise the PDFium Emscripten module.
  - Implement all `PdfDocumentImpl` methods (matching the native backend API
    surface) using Module C-function calls and WASM heap memory management.
  - v1 runs synchronously on the browser main thread; the plan records the
    Web Worker deferral explicitly.
- Update `spec/01_binary_distribution.md` to document the WASM artifact,
  schema extension, and distribution mechanism.
- Add a web integration test path (Flutter web or `dart2wasm` headless) to
  the test matrix.

**Prerequisite:** Windows support (`windows-x64` in the manifest and tooling)
should land first so the `version_pdfium.json` schema changes for the dual-file
WASM entry can be reviewed in context.

## Follow-on work items

These were identified during the WASM implementation review. The first two
below are complete; the third was discovered while implementing them and is
deferred to the Web Worker offload plan.

### Web backend: zero-copy WASM heap reads

**Status: Complete.**

**File:** `packages/betto_pdfium/lib/src/document/_document_web.dart`

Two sites on the render hot-path (page rendering, image rendering) called
`.toDart.sublist(ptr, ptr + byteCount)`, which allocated a full view of the
WASM heap (`Uint8List` of several MB) and then copied a slice out of it. This
was unnecessary allocation on every render call. Fixed by replacing with
`Uint8List.view(module.heapu8.toDart.buffer, bufPtr, byteCount)`, a zero-copy
view directly into the WASM `ArrayBuffer`. A third `.sublist` occurrence, in
`_readFileIdentifier`, was deliberately left unchanged — that buffer is freed
immediately after the read, so a zero-copy view there would alias memory the
allocator could reuse before the caller reads it.

### Web backend: unify bitmap format/stride conversion

**Status: Complete.**

**File:** `packages/betto_pdfium/lib/src/document/_document_web.dart`
**Related:** `packages/betto_pdfium/lib/src/document/_bitmap_utils.dart`

`_readThumbnailBitmap` hand-rolled BGR/BGRx/BGRA→compact BGRA pixel-by-pixel
conversion instead of delegating to `stripBitmapStride`, and the native
backend's `_handleGetPageThumbnail` (`pdfium_isolate.dart`) duplicated the same
loop. Fixed by extracting a shared `convertBitmapToCompactBgra(src, width,
height, stride, format, {srcOffset})` helper in `_bitmap_utils.dart` that
handles both stride removal and format conversion; both backends now call it,
eliminating the duplicated pixel loop.

### Web coverage gate: known gap, deferred to Web Worker offload plan

**Status: Complete.**

While implementing the two items above, `make web_coverage` was found to be
measuring the wrong thing entirely: the browser-based lcov report included
every script Chrome instrumented on the test page — transitive third-party
dependencies bundled into the same compiled JS (`package:test`,
`package:async`, `package:collection`, etc.) — none of which the Makefile
filtered out, unlike the VM-based `coverage` target's `*/generated/*`
exclusion. This made the reported number (39%) both wrong and misleadingly
low for the wrong reason.

Fixed:
- `betto_pdfium.mk`'s `web_coverage` target now runs `lcov --extract
  '*/betto_pdfium/lib/*'` before computing the percentage, narrowing the
  report to this package's own source.
- `web_test`/`web_coverage` were only ever running
  `test/pdf_document_web_test.dart`. Four platform-agnostic shared test files
  (`bitmap_util_test.dart`, `pdf_types_test.dart`, `pdf_date_parser_test.dart`,
  `pdf_page_size_test.dart` — none import `dart:io`/`dart:ffi`) never
  contributed to the web coverage number despite testing real web-relevant
  code. They now run under `-p chrome` alongside the dedicated web suite,
  which required fixing two genuine cross-platform test assertions
  (`pdf_page_size_test.dart`, `pdf_types_test.dart`) that assumed `595.0`-style
  double formatting — dart2js renders whole-number doubles without the
  trailing `.0`, unlike the Dart VM.

With both fixes, the *correctly measured* web coverage baseline was **83%**
(up from the previously-misleading 39%), still below the 90% gate. The
remaining gap was real but narrow — mostly additional annotation-subtype
fixtures needed in `test/pdf_document_web_test.dart` (e.g. the popup-linking
`switch` in `_document_web.dart` around lines 1579–1646 only exercised one or
two of six annotation subtypes) — not a measurement artifact.

Closed by the Web Worker offload plan's Phase 4
(`test/pdfium_wasm_engine_test.dart` plus several previously-generated-but-
unused annotation fixtures wired into both web test files). `make
web_coverage` now reports 92.8%, comfortably above the 90% gate.

### Web backend: Web Worker offload

**File:** `packages/betto_pdfium/lib/src/document/_document_web.dart`

**Status: Complete.**

> Plan at [`../plans/completed/plan_wasm_web_worker_offload.md`](../plans/completed/plan_wasm_web_worker_offload.md).

v1 of the web backend ran PDFium synchronously on the browser main thread —
every call blocked until it completed, which could freeze the tab during
large or complex document operations. `plan_wasm_support.md` explicitly
deferred a Web Worker offload path to a later release; this was the item
that let `README.md` drop the "(beta)" qualifier on Web (WASM) platform
support.

Research against the Flutter and Dart docs confirmed `dart:isolate` is not
usable on web at all (regardless of compile target), so the implementation
uses a hand-rolled `Worker` + `postMessage` protocol — the PDFium marshalling
engine was extracted into a worker-reusable module, a dedicated Worker entry
point is pre-compiled via a new maintainer-only `make build_wasm_worker`
target into a checked-in `lib/assets/pdfium_worker.js` artifact, and
`_document_web.dart`'s `PdfDocumentImpl` was rewired into a thin RPC client.
See the plan and `spec/02_pdfium_isolate.md`'s "Web Worker concurrency model"
section for the full architecture.
