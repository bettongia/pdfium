# Annotation Extraction

**Status**: Complete

**PR link**: _pending_

## Problem statement

`betto_pdf` can extract text and metadata from PDFs but has no annotation
support. Annotations (highlights, sticky notes, free-text comments, underlines,
strikethroughs, etc.) are a primary reading-workflow artefact and are explicitly
called out in the v0.02 roadmap. This plan covers:

1. **Dart library** — new public types and a `PdfDocument.extractAnnotations()`
   method backed by `fpdf_annot.h` FFI calls running on the PDFium isolate.
2. **CLI demo** — a `--annot` flag for `bin/pdfinfo.dart` that prints/JSON-encodes
   annotations alongside the existing metadata and text output.
3. **Python validation script** — `scripts/extract_annotations.py` (using
   `pypdf`) produces `.annot.json` reference files, mirroring the pattern of
   `extract_text.py` and `extract_meta.py`.
4. **Test fixtures** — new PDFs with known annotations generated by
   `test/fixtures/generate/generate_fixtures.py`.
5. **Tests** — ≥ 90 % coverage across all new code paths, including edge cases.

## Open questions

- [x] **Popup linkage**: `FPDF_ANNOT_POPUP` annotations are children of their
  parent sticky note / free-text annotation; should they be exposed as a
  separate top-level entry or inlined as a `popup` sub-object on the parent?
  _Decision_: inline as an optional `PdfPopupAnnotation` field on the
  parent to avoid confusing callers.
- [x] **Link annotations**: `FPDF_ANNOT_LINK` entries carry URI/page-destination
  actions but no visible mark-up. Include them in the extraction output or
  silently skip them? _Decision_: include with a `uri`/`destination`
  field and let callers filter; links are often the most machine-readable
  annotation type.
- [x] **Widget / form annotations**: `FPDF_ANNOT_WIDGET` and
  `FPDF_ANNOT_XFAWIDGET` are form fields, not reader annotations. Skip for now
  or expose read-only values? _Decision_: skip; form extraction is a separate
  roadmap item. A note will be added to the spec (`docs/spec/`) explicitly
  documenting that widget/form annotations are out of scope for v0.02 and
  earmarked for a future plan.
- [x] **pypdf annotation fidelity**: pypdf's annotation API is less complete
  than PDFium's. Confirm which fields pypdf can reliably extract for reference
  JSON, and document any fields that will differ between the two outputs
  (expected divergence, not a bug). _Decision_: the Python script will include
  a prominent module-level comment table listing every field that pypdf cannot
  reliably provide (e.g. ink paths, quad-point count, interior colour, popup
  linkage). For each such gap the corresponding `.annot.json` field will be
  `null`. Tests that rely on those fields must be written as Dart-only tests
  against the fixture PDFs directly, using PDFium as the sole source of truth —
  no pypdf reference to cross-check against.

## Investigation

### PDFium API surface (`fpdf_annot.h`)

All functions in `fpdf_annot.h` are marked `// Experimental API.` — the
abstraction layer is therefore critical.

**Page-level iteration:**

```c
int   FPDFPage_GetAnnotCount(FPDF_PAGE page);
FPDF_ANNOTATION FPDFPage_GetAnnot(FPDF_PAGE page, int index);
void  FPDFPage_CloseAnnot(FPDF_ANNOTATION annot);
```

**Per-annotation getters:**

| Getter | Notes |
|--------|-------|
| `FPDFAnnot_GetSubtype()` | Returns one of the `FPDF_ANNOT_*` constants |
| `FPDFAnnot_GetStringValue(annot, key, buf, len)` | Key strings: `"Contents"`, `"T"` (author), `"NM"` (name/id) |
| `FPDFAnnot_GetRect()` | Bounding rect in PDF page coordinates (bottom-left origin) |
| `FPDFAnnot_GetAttachmentPoints(annot, quadIndex, quadPoints)` | Quad-points for text markup (highlight, underline, etc.) |
| `FPDFAnnot_GetColor(annot, type, R, G, B, A)` | type = `FPDFANNOT_COLORTYPE_Color` or `_InteriorColor` |
| `FPDFAnnot_GetDate(annot, key)` | Key: `"M"` (modified date) |
| `FPDFAnnot_GetFlags()` | Bit-field of `FPDF_ANNOT_FLAG_*` |
| `FPDFAnnot_GetVertices()` | Polygon/polyline geometry |
| `FPDFAnnot_GetInkListPath()` | Ink (free-draw) paths |
| `FPDFAnnot_GetLine()` | Line annotation start/end points |
| `FPDFAnnot_GetBorder()` | Horizontal/vertical corner radii, border width |

**Annotation subtypes in scope for v0.02:**

| Subtype | Constant | Data of interest |
|---------|----------|-----------------|
| Sticky note | `FPDF_ANNOT_TEXT` | rect, contents, author, colour, date |
| Free text | `FPDF_ANNOT_FREETEXT` | rect, contents, author, colour |
| Highlight | `FPDF_ANNOT_HIGHLIGHT` | quad-points, colour, contents, author |
| Underline | `FPDF_ANNOT_UNDERLINE` | quad-points, colour, contents, author |
| Squiggly | `FPDF_ANNOT_SQUIGGLY` | quad-points, colour |
| Strikeout | `FPDF_ANNOT_STRIKEOUT` | quad-points, colour |
| Square | `FPDF_ANNOT_SQUARE` | rect, colour, interior colour |
| Circle | `FPDF_ANNOT_CIRCLE` | rect, colour, interior colour |
| Line | `FPDF_ANNOT_LINE` | start/end points, colour |
| Ink (free-draw) | `FPDF_ANNOT_INK` | ink paths, colour |
| Polygon | `FPDF_ANNOT_POLYGON` | vertices, colour |
| Polyline | `FPDF_ANNOT_POLYLINE` | vertices, colour |
| Link | `FPDF_ANNOT_LINK` | rect, URI or page-dest action |
| Popup | `FPDF_ANNOT_POPUP` | inlined onto parent (see open questions) |
| Stamp | `FPDF_ANNOT_STAMP` | rect, contents |

**Out of scope:** `FPDF_ANNOT_WIDGET`, `FPDF_ANNOT_XFAWIDGET`,
`FPDF_ANNOT_FILEATTACHMENT`, `FPDF_ANNOT_SOUND`, `FPDF_ANNOT_MOVIE`,
`FPDF_ANNOT_SCREEN`, `FPDF_ANNOT_REDACT`, `FPDF_ANNOT_WATERMARK`,
`FPDF_ANNOT_THREED`, `FPDF_ANNOT_RICHMEDIA`.

### FFI bindings strategy

`fpdf_annot.h` is **not yet included** in the `ffigen` configuration. The
header exists at `third_party/pdfium/public/fpdf_annot.h`. Steps:

1. Add `fpdf_annot.h` to the `ffigen` config (mirrors how `fpdf_text.h` was
   added for text extraction).
2. Run `make ffi_bindings` to regenerate
   `lib/src/generated/pdfium_bindings.dart`.
3. Commit the regenerated bindings alongside the plan implementation.

### Isolate message pattern

Follows the existing `PdfiumExtractPageTextCommand` / `PdfiumExtractPageTextResponse`
pair in `lib/src/document/isolate_messages.dart`. For annotations the
per-page approach is a good fit — one command per page, one response carrying
a `List<PdfAnnotation>`.

```
PdfiumExtractPageAnnotationsCommand { docToken, pageIndex }
PdfiumExtractPageAnnotationsResponse { pageIndex, annotations, error? }
```

The isolate handler in `_document_native.dart` opens the page, iterates
`FPDFPage_GetAnnotCount` / `FPDFPage_GetAnnot`, extracts fields, closes each
annotation handle, then closes the page handle.

### Public API shape

New types live in `lib/src/document/pdf_types.dart`:

**Design decision:** A sealed class hierarchy is used instead of a flat struct.
Each annotation subtype exposes only the fields relevant to it as non-nullable
members, giving callers compile-time guidance rather than a bag of nullable
extras. Common fields (`pageIndex`, `contents`, `author`, `rect`, `color`,
`modifiedDate`, `flags`) live on the sealed base class `PdfAnnotation`.

```dart
/// Base class for all annotation types.
sealed class PdfAnnotation {
  final int pageIndex;
  final String? contents;
  final String? author;
  final PdfRect? rect;
  final PdfColor? color;
  final PdfDate? modifiedDate;   // parsed via pdf_date_parser.dart
  final int flags;               // raw FPDF_ANNOT_FLAG_* bitmask
}

final class PdfTextAnnotation extends PdfAnnotation { /* sticky note */ }

final class PdfFreeTextAnnotation extends PdfAnnotation {}

final class PdfMarkupAnnotation extends PdfAnnotation {
  // covers highlight, underline, squiggly, strikeout
  final PdfAnnotationType subtype;   // one of the four markup subtypes
  final List<PdfQuadPoints> quadPoints;
}

final class PdfShapeAnnotation extends PdfAnnotation {
  // covers square, circle
  final PdfAnnotationType subtype;
  final PdfColor? interiorColor;
}

final class PdfLineAnnotation extends PdfAnnotation {
  final PdfPoint lineStart;
  final PdfPoint lineEnd;
}

final class PdfInkAnnotation extends PdfAnnotation {
  /// Each inner list is one stroke; each element is a point along that stroke.
  final List<List<PdfPoint>> strokes;
}

final class PdfPolygonAnnotation extends PdfAnnotation {
  final PdfAnnotationType subtype;   // polygon or polyline
  final List<PdfPoint> vertices;
}

final class PdfLinkAnnotation extends PdfAnnotation {
  final String? uri;
  // null when the link targets a page destination rather than a URI
}

final class PdfStampAnnotation extends PdfAnnotation {}

final class PdfUnknownAnnotation extends PdfAnnotation {
  final int rawSubtype;   // raw FPDF_ANNOT_* integer for debugging
}
```

Supporting value types (with `==`, `hashCode`, `toString`):

```dart
final class PdfRect { final double left, bottom, right, top; }
final class PdfPoint { final double x, y; }
final class PdfQuadPoints { /* 4 corner points of one quad */ }
final class PdfColor { final int r, g, b, a; }
```

`PdfAnnotationType` is retained as an enum for use on `PdfMarkupAnnotation` and
`PdfPolygonAnnotation` subtypes:

```dart
enum PdfAnnotationType {
  text, freeText, highlight, underline, squiggly, strikeout,
  square, circle, line, ink, polygon, polyline, link, stamp, popup, unknown,
}
```

`PdfDocument` gains one new method:

```dart
/// Extracts all annotations from every page as a stream, one page at a time.
///
/// Pages with no annotations still emit an entry with an empty [annotations]
/// list so callers can track page coverage.
///
/// An optional [pageIndex] restricts output to a single page, following the
/// same pattern as [extractPlainText].
///
/// Calling [close] while this stream is active terminates the stream
/// immediately and releases all page-level handles, consistent with the
/// guarantee provided by [extractPlainText].
///
/// **Note:** `extractAnnotations()` does not accept a page range — only a
/// single optional page index. Range-based extraction is a known limitation
/// to revisit in a future plan.
Stream<PdfPageAnnotations> extractAnnotations({int? pageIndex});
```

Where `PdfPageAnnotations` is:

```dart
final class PdfPageAnnotations {
  final int pageIndex;
  final List<PdfAnnotation> annotations;
}
```

### Coordinate system note

PDFium uses a bottom-left page origin for `PdfRect`. The plan exposes the raw
PDF coordinate space without conversion; callers that need screen coordinates
must apply `FPDF_PageToDevice()` themselves. This is consistent with how the
text layer currently reports character bounds.

### CLI additions (`bin/pdfinfo.dart`)

New `--annot` flag (mirrors `--text`):

- Plain text output: per-page section listing each annotation with its type,
  bounding rect, contents, author, colour, and any type-specific extras.
- JSON output: `"annotations"` key alongside the existing `"documentProperties"`,
  `"metadata"`, and optional `"text"` keys; value is an array of page objects
  matching `PdfPageAnnotations`.

### Python validation script (`scripts/extract_annotations.py`)

Follows the structure of `extract_text.py`:

- Uses `pypdf`'s `page.annotations` list (available in pypdf ≥ 3.x).
- Writes `.annot.json` alongside each PDF, containing an array of page objects
  `{ pageIndex, annotations: [{ type, rect, contents, author, color, … }] }`.
- Supports the same invocation patterns: no args (all PDFs under `test/data/`),
  file paths, or directory paths.
- Documents which fields pypdf does/does not expose, so expected divergence
  from the Dart output is explicit.

pypdf's annotation support is limited: `page.annotations` returns raw
`DictionaryObject` entries. The script will extract `/Subtype`, `/Contents`,
`/T` (author), `/Rect`, `/QuadPoints`, `/C` (colour), and `/M` (date) directly
from the dictionary. Fields that PDFium exposes but pypdf cannot (e.g.
`FPDF_ANNOT_GetAttachmentPoints` quad-point count, ink paths) will be `null`
in the reference JSON, which is acceptable for validation purposes.

### Edge cases for tests

- Page with zero annotations — should return empty `annotations` list, not error.
- Annotation with no `Contents` field — `contents` is `null`. An annotation
  whose `Contents` entry exists but is an empty string — `contents` is `""`.
  The two cases are intentionally distinguishable.
- Annotation with no colour entry — `color` is `null`.
- Annotation with malformed `/Rect` entry — should not crash; return `rect: null`.
- Annotation with quad-points count not a multiple of 8 — gracefully truncate.
- `FPDF_ANNOT_UNKNOWN` subtype — map to `PdfAnnotationType.unknown`, preserve
  the raw integer in a debug field.
- Popup annotations — inlined on parent (no separate top-level entry).
- Concurrent `extractAnnotations()` calls — permitted. The isolate handles
  concurrent annotation streams via per-request `SendPort`s, the same
  mechanism used for concurrent `extractPlainText()` calls.
- Very large annotation counts — no artificial limit. If a page with a large
  number of high-point-count ink paths causes issues in practice, that will be
  addressed when the problem arises rather than pre-emptively.
- Document with no pages — `extractAnnotations()` stream completes immediately.
- Password-protected PDF — raises `PdfExtractionException` (existing behaviour).

## Reviews

### Review 1: 2026-05-19

_Reviewed: 2026-05-19_

**Problem Statement Assessment**

The problem is real, well-scoped, and correctly positioned in the v0.02 roadmap.
Annotations are a first-class reading-workflow artefact and the explicit roadmap
entry confirms this work is planned. The scope is appropriately bounded: reading
only, no write-back, no form/widget fields, no multimedia subtypes. No concerns
with the problem itself.

**Proposed Solution Assessment**

The overall approach is sound. It follows the established isolate-message pattern,
adds a new public stream method consistent with `extractPlainText()`, and produces
fixtures + a Python validation script in keeping with the project convention.
There are, however, several concrete issues that need attention before
implementation.

1. **`modifiedDate` type inconsistency — the most important issue.** The existing
   codebase parses PDF date strings into `PdfDate` (raw string + `DateTime?`)
   using `pdf_date_parser.dart`. The proposed `PdfAnnotation` stores
   `modifiedDate` as a raw `String?` instead. This is an inconsistency that will
   either confuse callers (two date representations in the same library) or force
   a follow-up fix. The field should be typed as `PdfDate?` and parsed via the
   existing `pdf_date_parser.dart`, consistent with `PdfMetadata.modDate` and
   `PdfMetadata.creationDate`.

2. **`PdfAnnotation` is a flat struct for a type-union problem.** The design
   puts `inkPaths`, `vertices`, `lineStart`, `lineEnd`, `linkUri`, and
   `quadPoints` all on one class. For most annotation types most of these fields
   are `null`. While this pattern is workable, it creates ambiguity: callers have
   no compile-time guidance about which fields are populated for a given subtype.
   A sealed class hierarchy (`PdfTextAnnotation`, `PdfHighlightAnnotation`,
   `PdfInkAnnotation`, `PdfLinkAnnotation`, etc.) would be more expressive and
   would make `type`-specific fields non-nullable on the appropriate subclass.
   This is a genuine design trade-off; the flat struct is simpler to serialize
   and the plan's precedents (query: is this how text is handled?) are on its
   side. However, the plan should explicitly acknowledge the trade-off and record
   why a sealed hierarchy was rejected — currently it simply asserts the flat
   struct without justification.

3. **`FPDF_ANNOT_POPUP` parent-linking is unspecified at the FFI level.** The
   plan says popup annotations should be inlined on the parent, but it does not
   describe how to find that parent. PDFium exposes `FPDFAnnot_GetLinkedAnnot()`
   for this purpose (available on `FPDF_ANNOT_POPUP` to retrieve the parent, and
   on sticky-note/free-text to retrieve their popup). This function needs to be
   in the FFI binding surface and the isolate handler needs to perform a
   two-pass iteration: first pass collects all non-popup annotations keyed by
   native handle index; second pass matches popups to parents. The current plan
   provides no detail here. Missing detail at this level will cause guesswork
   during implementation.

4. **`PdfRect` name conflict risk.** The plan introduces `PdfRect` as a new
   value type. There is currently no `PdfRect` in `pdf_types.dart`, so there is
   no direct collision — but the Flutter SDK also has a `Rect` class, and
   `pdf_document.dart` imports `dart:ui` on the native side. Review imports
   carefully to ensure no shadowing. This is low-risk but worth an explicit note.

5. **Web / stub stubs are mentioned but not described.** Phase 5 says "update
   `_document_stub.dart` and `_document_web.dart` stubs" without describing what
   those stubs should do. Based on the existing pattern for `extractPlainText()`,
   the stub should throw `UnsupportedError` and the web stub should return a
   stream that immediately throws `UnsupportedError`. The plan should state this
   explicitly so the implementer does not accidentally return an empty stream
   (which would silently appear to work but return no data).

6. **No `close()` interaction described for annotation streams.** The doc
   comment for `extractPlainText()` explicitly states that `close()` terminates
   any active stream. The plan does not address whether `extractAnnotations()`
   has the same guarantee. It should — and the implementation plan should
   reference it. Failure to handle this results in a handle leak if the caller
   calls `close()` while an annotation stream is in progress.

7. **`inkPaths` type is `List<PdfRect>?`, but ink paths are sequences of
   points.** PDFium's `FPDFAnnot_GetInkListPath()` returns a sequence of
   `FS_POINTF` structures (x/y pairs), not rectangles. The type should be
   `List<List<PdfPoint>>?` — a list of strokes, each stroke being a list of
   points. Using `PdfRect` here is semantically wrong and will produce incorrect
   data.

8. **`extractAnnotations()` always streams all pages; no per-page or range
   option.** `extractPlainText()` accepts an optional `pageIndex` parameter.
   Given that real documents may have annotations only on a few pages, a similar
   `pageIndex` parameter would be useful for callers that know which page they
   want. This is a minor omission — it can be deferred — but the plan should
   note the limitation explicitly so it is not discovered post-release.

9. **fpdf2 fixture annotation fidelity.** The plan relies on fpdf2 for fixture
   PDF generation. fpdf2's annotation API differs from PDF readers (e.g. fpdf2
   stores quad-points in a specific order; its color model is RGB 0–255). The
   plan says "verify the required fpdf2 API before implementing" but provides no
   detail. At minimum the plan should state the expected fpdf2 version constraint
   (the plan notes v2.7.x) and identify which annotation types fpdf2 supports
   for generation — fpdf2's free-draw ink annotation support in particular is
   limited. This is a non-blocking note but the implementer should verify ink
   fixture generation is achievable before committing to the `annotated_ink.pdf`
   fixture.

**Architecture Fit**

The plan fits the existing architecture well: isolate-message protocol, sealed
command/response pairs, per-page streaming, `PdfDocument` as the API surface,
and `pdf_types.dart` as the shared type file. The decision to keep FFI bindings
behind a Dart abstraction layer is correct given the Experimental API status of
`fpdf_annot.h`. The spec will need a new `annotation_extraction.md` section (the
plan mentions updating `docs/spec/` only to add the widget/form exclusion note,
which is insufficient — the new public API surface deserves a full spec entry
matching the depth of `text_extraction.md`).

**Risk & Edge Cases**

The edge-case list is good. Three gaps:

- **Annotation with a `Contents` value that is an empty string vs. absent** —
  _Decision_: empty string → `""`, absent → `null`. The two cases are
  intentionally distinguishable. Updated in plan body.
- **Concurrent `extractAnnotations()` calls** — _Decision_: concurrent streams
  are permitted, handled via per-request `SendPort`s as with
  `extractPlainText()`. Documented in edge cases.
- **Very large annotation counts** — _Decision_: no artificial limit imposed.
  If a page with a large number of high-point-count ink paths causes issues in
  practice, that will be addressed when the problem arises.
